    Jeff JettonSignature Track· a day ago 

    If you're using caret's "train" function, it helps to understand what it's really doing.

    Let's say we just do a straightforward random forest, like so:

    fit <- train(y ~ ., data=trainset, method="rf")

    Here's what's really happening behind-the-scenes, by default:

    1. Caret does a 25-fold cross-validation using random forest, with the "mtry" parameter (the number of variables randomly-selected to go in each tree) set to 2 and ntree set to the default of 500.
      • That's 500 trees being created for each of 25 different forests, for a total of 12,500 trees!
    2. Caret does another 25-fold cross-validation, but this time sets the mtry parameter to about half the number of training features.
    3. One more round, but with mtry set to the full number of training features.

    So it's not just creating one model. It's essentially creating 75 different random forests, each with 500 trees, for a total 37,500 classification trees! And in your case, that last batch of trees will be a full, 87-feature fit!

    No wonder it's running slowly!

    To speed things up, you could skip using caret's train function entirely and just call randomForest yourself directly, with your own choice of mtry and ntree (or just use the defaults). That will be about 75 times faster right there, but of course you won't have the accuracy estimate that cross-validation provides, and you won't be "tuning" any parameters.

    Another option is to still use caret's train function, but override the defaults. For example, this will change the cross-validation to 5-fold instead of the default of 25-fold:

    fit <- train(y ~ ., data=trainset, method="rf", trControl=traincontrol(number=5))

    And this is how you might choose a smaller number of training parameter variations:

    customGrid <- data.frame(mtry=c(5, 9))   # Test with mtry=5 and again with mtry=9
fit <- train(y ~ ., data=trainset, method="rf", trControl=traincontrol(number=5), tuneGrid=customGrid)

    Next, you could reduce the number of trees grown in each forest:

    fit <- train(y ~ ., data=trainset, method="rf", trControl=traincontrol(number=5),
             tuneGrid=customGrid, ntree=101)

    If you test this stuff on a small (like, 1/50 sized) subset of your data, you can get a sense of how these changes affect your accuracy. (The accuracy may be lower on a smaller set, but that's okay... we're more concerned with the relative differences in accuracy as we make our tweaks.)

    Reducing the number of folds will probably increase the Accuracy SD, but probably not by much. And the difference in accuracy you get for different mtry values might not be enough to bother with even tuning by it. Ditto for ntree. Experiment and see for yourself!

    You also might notice a speed up if you explicitly enable parallel processing, as talked about in the thread Stuart mentioned. I found that to work well on my Windows box, but I didn't notice much difference on my Mac (perhaps Macs are already taking advantage of multiple cores?)

    Jeff JettonSignature Track· 10 hours ago 

    Okay, more poking around and thinking. :-)

    By default, trainControl uses the "boot" (i.e. bootstrap) method to resample the data and gauge the accuracy of your particular model. The "number" parameter (default of 25) sets the number of times it dips into a new, randomly-bootstrapped data set, and "repeats" is apparently not even used.

    So if you kept the default trainControl method, you, in theory, should not have noticed any performance difference by changing "repeats". (Or, if you did, it was due to something else.)

    The online docs go into a bit more detail than the current R help file for trainControl: http://topepo.github.io/caret/training.html#control

    Setting the `verboseIter=TRUE' argument and doing some trains on a small dataset may be a worthwhile experiment.

    Now what's weird is that the code in trainControl is grepping the method and setting the "repeats" and "number" defaults based on whether "cv" is in the method name (case-sensitive, so it won't catch LOOCV, for example). The grep will return TRUE not only with "repeatedcv" but also with "cv".

    But the code in train (do a search for "repeats") looks like it truly is only using the value in "repeats" when "repeatedcv" or "adaptive_cv" is used.

    Which means that setting the trainControl method to "cv" will still change the default number of folds to 10, even though it doesn't bother with "repeats" value at all. (I tested this myself and it's true... "cv" only does 10 folds by default.) I wonder if that's a bug?

    Incidentally, the default behavior* for caret's "gbm" method is to test the model while varying two different tuning parameters: interaction.depth (varies from 1 to 3) and n.trees (50, 100, and 150 trees). That's nine different combinations, with ten cross-validations each (in your case), for a total of 90 gbm models being created.

    * Check out the source code at lines 11-14. That bit of code creates the default search grid of testing parameters when gbm is used. The len variable comes from the call to the train function, where it is set, by default, to 3.
